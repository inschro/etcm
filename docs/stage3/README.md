# Stage 3: Fixture Contract And Parser Gates

Stage 3 is the final planning checkpoint before parser implementation. It turns
the product semantics, Stage 1 tool decisions, and Stage 2 scaffold into a
fixture and golden-output contract that future code must satisfy.

The purpose is to remove guesswork from Stage 4. The parser should be built
against named examples, stable diagnostics, and deterministic golden formats,
not against an implicit reading of the manifest.

## Outcome

Stage 3 should leave the project with:

- a complete v0 fixture matrix
- a documented golden-output format for AST, IR, resolver, generated views, and
  diagnostics
- a diagnostic taxonomy with stable codes and required fields
- parser entry gates that separate syntax work from resolver and codegen work
- explicit reasoning for why CLI and resolver behavior remain out of scope

## Non-Goals

Stage 3 does not implement:

- production parser behavior
- resolver semantics
- graph construction
- generated Pydantic views
- CLI commands
- migration tooling

Stage 3 also does not require final golden files generated by implementation
code. Where a serializer does not exist yet, Stage 3 defines the intended shape
instead of fabricating output that future code may accidentally treat as truth.

## Exit Criteria

Stage 3 is complete when:

- every v0 behavior has at least one planned valid or invalid fixture
- every planned failure category has a diagnostic code and required fields
- golden outputs have deterministic ordering and path normalization rules
- parser-only Stage 4 work can proceed without making resolver or CLI decisions
- Stage 2 docs no longer imply parser behavior belongs in Stage 3
